TDD - сперва пишем тест, который фейлится, затем код, который делает этот тест зелёным.

Три правила tdd:
- не пишем готовый код до того, как напишем готовый тест, который не получается пройти из-за нехватки этого кода 
- не пишем тестов больше, чем необходимо для неудачи - сбой при компиляции тоже считается неудачей 
- не пишем готового кода больше, чем достаточно для прохождения теста, который был провален до этого. 

Полнота тестов - это показатель для команды, а не для менеджеров. 

Рефакторинг - это метод улучшения структуры кода без изменения его поведения, определённого тестами. 

Он не проходит по расписанию, это часть ежедневной рутины при работе с кодом. 

Цикл красный - зелёный - рефакторинг - пишем тест, который не проходит, потом пишем код, который зеленит тест, 
апотом подчищаем код, и возвращается к началу цикла, пишем следующий тест. 

Простое проектирование состоит в:
- пройти все тесты
- проявить намерение 
- удалить дубликаты
- сократить количество элементов

Цель - сохранить равновесие между требованиями и достижением наиболее высокой производительности. 

Парное программирование - это совместная работа двух программистов над одной задачей. 

Плюсы - обмен опытом, код ревью, глубокое взаимодействие, укрепление командного духа. 
Минусы - над задачей работают два человека
